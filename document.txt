What is st.session_state?

    st.session_state is a special dictionary-like object provided by Streamlit.

    It stores values that persist across reruns of your Streamlit app.

    Normally, Streamlit runs your script top to bottom on every user interaction, so all variables reset.

    st.session_state lets you keep data alive between those reruns — like user inputs, flags, counters, etc.

How does it work internally?

    Streamlit runs your entire script on every interaction (like a button click or input).

    Before each run, Streamlit restores st.session_state from the browser/user session.

    When your script modifies st.session_state, Streamlit saves those changes back to the session.

    This way, data inside st.session_state persists across reruns and between widgets.

// to check cache eviction policy
    redis-cli CONFIG GET maxmemory-policy
// to check cache size
    redis-cli CONFIG GET maxmemory

maxmemory = 0 (Redis will use as much memory as the operating system allows. It won’t automatically start evicting keys unless you hit a system-level memory limit (e.g., the machine runs out of RAM).)
maxmemory-policy = noeviction (When Redis is full (or OS is out of memory), it will NOT evict any keys. Instead, it will start returning errors for commands that require memory allocation (e.g., SET, SETEX).)

Your cache will grow indefinitely (bounded only by system RAM).

No old keys will be removed automatically, unless you’ve set a TTL (which you did with setex, so they expire naturally after 60 seconds).

If you don’t set TTL, the data will stay forever and might cause memory issues if the cache usage grows.

to add config in redis.conf file (/opt/homebrew/etc/redis.conf) , we can set memory size and eviction policy then restart redis
maxmemory 512mb
maxmemory-policy allkeys-lru
